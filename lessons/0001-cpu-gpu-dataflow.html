2. Optimize the dominant term

frame cost =
capture + resize + preprocess + transfer +
inference + postprocess + render + output

On the current macOS CPU setup, pose inference is about 250–330 ms while resize and rendering are typically below 2 ms combined. Removing a one-millisecond stage cannot rescue a 250-millisecond model.

3. Practical Orin path

Stage A: accelerate inference

Use CUDA FP16, keep fresh inference on every frame, benchmark without display or saving, and confirm GPU load with tegrastats.

Stage B: replace the heavy model

ResizeProcessor
→ ONNX/TensorRT pose adapter
→ canonical Pose
→ tracking and analytics
→ SkeletonRenderer

Stage C: reduce transfer cost

camera / compressed video
→ hardware decode / NVMM
→ GPU preprocessing
→ TensorRT
→ small pose result returned to CPU

4. Throughput is not latency

Capture, inference, and output can eventually run in bounded asynchronous stages. Keep queues small and drop old frames when necessary; unlimited queues increase delay even if reported throughput looks good.
